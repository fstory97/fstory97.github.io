# AI 업무 방법 가이드 (For AI Assistance)

## 1. 개요

이 문서는 AI 어시스턴트가 작업을 수행하는 표준 절차와 원칙을 정의합니다. AI는 이 가이드라인을 숙지하고 모든 작업을 이 절차에 따라 수행해야 합니다.

**🚨 핵심 원칙**: **업무 성격별 필수 문서 체크 없이는 코딩 시작 금지**

## 2. 작업 착수 프로토콜 (강화된 체크포인트)

개발자로부터 작업 지시를 받으면, AI는 다음 단계를 **반드시 순서대로** 수행해야 합니다. 코딩 또는 파일 수정을 시작하기 전에 이 프로토콜을 완료하는 것은 **매우 중요합니다 (CRITICAL)**.

3. **작업 로그 확인**: `caret-docs/work-logs/{username}/{date}.md` 체크/생성

## 3. 올바른 TDD 원칙 (⚠️ 흔한 실수 방지)

### 🚨 **TDD 흔한 실수: "거꾸로 TDD"**

**❌ 잘못된 접근 (단위 테스트부터)**:
1. 단위 함수들의 테스트 작성 → 단위 함수 구현 → 리팩터
2. 나중에 "실제 사용처에 통합"

**✅ 올바른 TDD 접근 (통합 시나리오부터)**:
1. **RED**: 실제 사용 시나리오의 **통합 테스트**부터 작성 (E2E 또는 Integration)
2. **GREEN**: 통합 테스트가 통과하도록 필요한 모든 코드 구현 (단위 함수들 포함)
3. **REFACTOR**: 코드 품질 개선 (테스트는 계속 통과)

### 🎯 **올바른 TDD 시작점 결정**

#### **웹뷰 기능 개발**
- ❌ 잘못: `isBackendMessage()` 단위 함수 테스트부터
- ✅ 올바름: "웹뷰에서 ClineMessage 렌더링 시 백엔드 메시지가 번역되어야 함" 통합 테스트부터

#### **API/서비스 개발**  
- ❌ 잘못: `validateRequest()` 단위 함수 테스트부터
- ✅ 올바름: "HTTP 요청 → 응답" 전체 흐름 통합 테스트부터

#### **UI 컴포넌트 개발**
- ❌ 잘못: `formatText()` 유틸리티 테스트부터  
- ✅ 올바름: "컴포넌트 렌더링 → 사용자 상호작용 → 기대 결과" 컴포넌트 테스트부터

### **Phase 1: TDD RED - 통합 테스트 우선 작성**

**🛑 STOP POINT 1: 실제 사용 시나리오 테스트 작성**

1. **실제 사용처부터 시작**: 최종 사용자 관점에서 테스트 작성
2. **End-to-End 흐름**: 입력 → 처리 → 출력 전체 시나리오  
3. **실패 확인**: 테스트가 RED (실패) 상태인지 확인
4. **필요한 인터페이스 설계**: 테스트 작성 과정에서 필요한 함수/클래스 인터페이스 도출

**예시**:
```typescript
// ✅ 올바른 시작: 실제 사용 시나리오부터
test('웹뷰에서 백엔드 메시지가 한국어로 번역되어야 함', () => {
  const message = { text: "Cline wants to edit file.js", type: "say" }
  const rendered = renderMessage(message, { mode: 'caret', language: 'ko' })
  expect(rendered).toContain('Caret이 파일을 편집하려고 합니다')
})
```

### **Phase 2: TDD GREEN - 통합 테스트 통과시키기**

**🛑 STOP POINT 2: 최소한의 구현으로 통합 테스트 통과**

1. **통합 테스트부터 통과**: 단위 테스트가 아닌 통합 테스트가 GREEN이 되도록
2. **필요한 코드만**: 통합 테스트 통과에 필요한 최소한의 코드만 구현
3. **단위 함수는 부산물**: 통합 테스트 통과 과정에서 자연스럽게 생성되는 단위 함수들
4. **즉시 검증**: 각 구현 후 통합 테스트 실행으로 진행상황 확인

### **Phase 3: TDD REFACTOR - 코드 품질 개선**

**🛑 STOP POINT 3: 통합 테스트를 유지하며 리팩터링**

1. **통합 테스트 유지**: 리팩터링 중에도 통합 테스트는 계속 통과
2. **단위 테스트 추가**: 필요시 복잡한 단위 함수에 대한 단위 테스트 추가 (선택사항)
3. **코드 품질 개선**: 가독성, 유지보수성, 성능 개선
4. **최종 검증**: 모든 테스트 통과 및 기능 동작 확인

### **🔄 TDD 사이클 체크리스트**

- [ ] **1. RED**: 실제 사용 시나리오의 통합/E2E 테스트 작성 (실패 확인)
- [ ] **2. GREEN**: 통합 테스트가 통과하도록 최소 구현  
- [ ] **3. REFACTOR**: 통합 테스트를 유지하며 코드 품질 개선
- [ ] **4. 반복**: 다음 시나리오로 사이클 반복

2. **패턴 기반 구현**:
    - 분석한 아키텍처 패턴 적용
    - 기존 구현 사례 참조
    - 일관성 있는 구현 방식 유지

3. **실시간 문서 업데이트**:
    - 작업 체크리스트와 보고서 실시간 업데이트
    - 새로운 패턴 발견 시 관련 문서 업데이트 제안
    - 문제 해결 과정 문서화

4. **검증 및 테스트**:
    - 각 Phase 완료 후 검증
    - 관련 테스트 실행 및 통과 확인
    - 기존 기능 회귀 없음 확인

## 4. 금지 사항 ❌

1. **즉시 코딩 시작 금지**: 관련 문서 체크 없이 바로 코드 수정 시작

3. **테스트 생략 금지**: 구현 후 테스트 작성하는 잘못된 순서
4. **문서 업데이트 생략 금지**: 새로운 패턴 발견 시 문서화 생략
