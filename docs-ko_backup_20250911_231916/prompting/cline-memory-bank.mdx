### Cline 메모리 뱅크 사용자 지정 지침 \[이것을 복사하세요]

```
# Cline의 메모리 뱅크

저는 Cline입니다. 세션 간에 기억이 완전히 재설정되는 독특한 특성을 가진 전문 소프트웨어 엔지니어입니다. 이것은 한계가 아니라 완벽한 문서를 유지하도록 저를 이끄는 원동력입니다. 각 재설정 후, 저는 프로젝트를 이해하고 작업을 효과적으로 계속하기 위해 전적으로 메모리 뱅크에 의존합니다. 저는 모든 작업 시작 시 모든 메모리 뱅크 파일을 읽어야 합니다. 이것은 선택 사항이 아닙니다.

## 메모리 뱅크 구조

메모리 뱅크는 핵심 파일과 선택적 컨텍스트 파일로 구성되며, 모두 마크다운 형식입니다. 파일은 명확한 계층 구조로 서로를 기반으로 구축됩니다.

flowchart TD
    PB[projectbrief.md] --> PC[productContext.md]
    PB --> SP[systemPatterns.md]
    PB --> TC[techContext.md]

    PC --> AC[activeContext.md]
    SP --> AC
    TC --> AC

    AC --> P[progress.md]

### 핵심 파일 (필수)
1. `projectbrief.md`
   - 다른 모든 파일을 형성하는 기본 문서
   - 프로젝트 시작 시 존재하지 않으면 생성됨
   - 핵심 요구 사항 및 목표 정의
   - 프로젝트 범위에 대한 진실의 원천

2. `productContext.md`
   - 이 프로젝트가 존재하는 이유
   - 해결하는 문제
   - 작동 방식
   - 사용자 경험 목표

3. `activeContext.md`
   - 현재 작업 초점
   - 최근 변경 사항
   - 다음 단계
   - 활성 결정 및 고려 사항
   - 중요한 패턴 및 기본 설정
   - 학습 및 프로젝트 통찰력

4. `systemPatterns.md`
   - 시스템 아키텍처
   - 주요 기술적 결정
   - 사용 중인 디자인 패턴
   - 구성 요소 관계
   - 중요한 구현 경로

5. `techContext.md`
   - 사용된 기술
   - 개발 설정
   - 기술적 제약
   - 종속성
   - 도구 사용 패턴

6. `progress.md`
   - 작동하는 것
   - 구축할 남은 것
   - 현재 상태
   - 알려진 문제
   - 프로젝트 결정의 진화

### 추가 컨텍스트
메모리 뱅크/ 내에 추가 파일/폴더를 생성하여 다음을 구성하는 데 도움이 되도록 합니다.
- 복잡한 기능 문서
- 통합 사양
- API 문서
- 테스트 전략
- 배포 절차

2. 중요한 변경 사항 구현 후
3. 사용자가 **메모리 뱅크 업데이트**를 요청할 때 (모든 파일을 검토해야 함)

기억하십시오: 모든 메모리 재설정 후, 저는 완전히 새로 시작합니다. 메모리 뱅크는 이전 작업에 대한 유일한 연결 고리입니다. 저의 효과는 전적으로 정확성에 달려 있으므로 정밀하고 명확하게 유지되어야 합니다.
```

### Cline 메모리 뱅크란 무엇인가요?

메모리 뱅크는 Cline이 세션 간에 컨텍스트를 유지할 수 있도록 하는 구조화된 문서 시스템입니다. Cline을 상태 비저장 비서에서 시간이 지남에 따라 프로젝트 세부 정보를 효과적으로 "기억"할 수 있는 영구적인 개발 파트너로 변모시킵니다.

#### 주요 이점

-   **컨텍스트 보존**: 세션 간에 프로젝트 지식 유지
-   **일관된 개발**: Cline과의 예측 가능한 상호 작용 경험
-   **자체 문서화 프로젝트**: 부수적으로 가치 있는 프로젝트 문서 생성
-   **모든 프로젝트에 확장 가능**: 모든 크기 또는 복잡성의 프로젝트에서 작동
-   **기술에 구애받지 않음**: 모든 기술 스택 또는 언어에서 작동

### 메모리 뱅크 작동 방식

메모리 뱅크는 Cline 특정 기능이 아니라 구조화된 문서를 통해 AI 컨텍스트를 관리하는 방법론입니다. Cline에게 "사용자 지정 지침을 따르라"고 지시하면 메모리 뱅크 파일을 읽어 프로젝트에 대한 이해를 재구축합니다.

파일은 프로젝트의 전체 그림을 구축하는 계층적 구조로 구성됩니다.

### 메모리 뱅크 파일 설명

#### 핵심 파일

1.  **projectbrief.md**
    -   프로젝트의 기반
    -   구축 중인 내용에 대한 상위 수준 개요
    -   핵심 요구 사항 및 목표
    -   예시: "바코드 스캐닝을 통한 재고 관리를 위한 React 웹 앱 구축"
2.  **productContext.md**
    -   프로젝트가 존재하는 이유 설명
    -   해결 중인 문제 설명
    -   제품 작동 방식 개요
    -   예시: "재고 시스템은 여러 창고와 실시간 업데이트를 지원해야 합니다."
3.  **activeContext.md**
    -   가장 자주 업데이트되는 파일
    -   현재 작업 초점 및 최근 변경 사항 포함
    -   활성 결정 및 고려 사항 추적
    -   중요한 패턴 및 학습 저장
    -   예시: "현재 바코드 스캐너 구성 요소 구현 중; 지난 세션에서 API 통합 완료"
4.  **systemPatterns.md**
    -   시스템 아키텍처 문서화
    -   주요 기술적 결정 기록
    -   사용 중인 디자인 패턴 나열
    -   구성 요소 관계 설명
    -   예시: "정규화된 저장소 구조로 상태 관리를 위해 Redux 사용"
5.  **techContext.md**
    -   사용된 기술 및 프레임워크 나열
    -   개발 설정 설명
    -   기술적 제약 기록
    -   종속성 및 도구 구성 기록
    -   예시: "React 18, TypeScript, Firebase, 테스트용 Jest"
6.  **progress.md**
    -   작동하는 것과 구축할 남은 것 추적
    -   기능의 현재 상태 기록
    -   알려진 문제 및 제한 사항 나열
    -   프로젝트 결정의 진화 문서화
    -   예시: "사용자 인증 완료; 재고 관리 80% 완료; 보고서 시작 안 함"

#### 추가 컨텍스트

필요할 때 추가 파일을 생성하여 다음을 구성합니다.

-   복잡한 기능 문서
-   통합 사양
-   API 문서
-   테스트 전략
-   배포 절차

3.  Cline에게 "메모리 뱅크 초기화" 요청

### Cline과 함께 작업하기

#### 핵심 워크플로

**계획 모드**

전략 논의 및 상위 수준 계획을 위해 이 모드에서 시작하십시오.

**실행 모드**

구현 및 특정 작업 실행을 위해 이 모드를 사용하십시오.

#### 주요 명령

-   **"사용자 지정 지침을 따르십시오"** - 이는 Cline에게 메모리 뱅크 파일을 읽고 중단된 지점에서 계속하도록 지시합니다(작업 시작 시 사용).
-   **"메모리 뱅크 초기화"** - 새 프로젝트를 시작할 때 사용
-   **"메모리 뱅크 업데이트"** - 작업 중에 전체 문서 검토 및 업데이트를 트리거합니다.
-   현재 필요에 따라 계획/실행 모드 전환

#### 문서 업데이트

메모리 뱅크 업데이트는 다음 경우에 자동으로 발생해야 합니다.

1.  프로젝트에서 새로운 패턴을 발견할 때
2.  중요한 변경 사항 구현 후
3.  **"메모리 뱅크 업데이트"**로 명시적으로 요청할 때
4.  컨텍스트에 대한 설명이 필요하다고 느낄 때

### 자주 묻는 질문

#### 메모리 뱅크 파일은 어디에 저장되나요?

메모리 뱅크 파일은 프로젝트 저장소에 저장된 일반 마크다운 파일이며, 일반적으로 `memory-bank/` 폴더에 있습니다. 숨겨진 시스템 파일이 아니라 프로젝트 문서의 일부로 설계되었습니다.

#### 사용자 지정 지침 또는 .clinerules를 사용해야 하나요?

두 가지 접근 방식 모두 작동하며, 이는 기본 설정에 따라 다릅니다.

-   **사용자 지정 지침**: 모든 Cline 대화에 전역적으로 적용됩니다. 모든 프로젝트에서 일관된 동작에 좋습니다.
-   **.clinerules 파일**: 프로젝트별이며 저장소에 저장됩니다. 프로젝트별 사용자 지정에 좋습니다.

두 가지 방법 모두 동일한 목표를 달성합니다. 선택은 메모리 뱅크 시스템의 전역 또는 로컬 적용 여부에 따라 달라집니다.

#### 컨텍스트 창 관리

Cline과 함께 작업할 때 컨텍스트 창은 결국 가득 찰 것입니다(진행률 표시줄 참고). Cline의 응답 속도가 느려지거나 대화의 이전 부분에 대한 참조가 덜 정확해지는 것을 발견하면 다음을 수행할 때입니다.

1.  Cline에게 **"메모리 뱅크 업데이트"**를 요청하여 현재 상태를 문서화합니다.
2.  새 대화/작업 시작
3.  새 대화에서 Cline에게 **"사용자 지정 지침을 따르십시오"**라고 요청합니다.

이 워크플로는 컨텍스트 창이 지워지기 전에 중요한 컨텍스트가 메모리 뱅크 파일에 보존되도록 하여 새로운 대화에서 원활하게 계속할 수 있도록 합니다.

#### 메모리 뱅크를 얼마나 자주 업데이트해야 하나요?

중요한 이정표 또는 방향 변경 후 메모리 뱅크를 업데이트하십시오. 활성 개발의 경우 몇 세션마다 업데이트하는 것이 도움이 될 수 있습니다. 모든 컨텍스트가 보존되도록 하려면 **"메모리 뱅크 업데이트"** 명령을 사용하십시오. 그러나 Cline이 메모리 뱅크를 자동으로 업데이트하는 것을 볼 수 있습니다.

#### 이것이 Cline 외의 다른 AI 도구에서도 작동하나요?

예! 메모리 뱅크 개념은 문서 파일을 읽을 수 있는 모든 AI 비서와 함께 작동할 수 있는 문서 방법론입니다. 특정 명령은 다를 수 있지만 컨텍스트를 유지하는 구조화된 접근 방식은 도구 전반에 걸쳐 작동합니다.

#### 메모리 뱅크는 컨텍스트 창 제한과 어떻게 관련되나요?

### 모범 사례

#### 시작하기

-   기본 프로젝트 브리프로 시작하고 구조가 발전하도록 합니다.
-   Cline이 초기 구조를 만드는 데 도움을 주도록 합니다.
-   워크플로에 맞게 파일을 검토하고 조정합니다.

#### 지속적인 작업

-   작업하면서 패턴이 자연스럽게 나타나도록 합니다.
-   문서 업데이트를 강요하지 마십시오. 유기적으로 발생해야 합니다.
-   프로세스를 신뢰하십시오. 가치는 시간이 지남에 따라 복합적으로 증가합니다.
-   세션 시작 시 컨텍스트 확인을 주시하십시오.

#### 문서 흐름

-   **projectbrief.md**는 당신의 기반입니다.
-   **activeContext.md**는 가장 자주 변경됩니다.
-   **progress.md**는 당신의 이정표를 추적합니다.
-   모든 파일은 함께 프로젝트 인텔리전스를 유지합니다.

### 상세 설정 지침

#### 사용자 지정 지침 (전역)

1.  VSCode 열기
2.  Cline 확장 설정 ⚙️ 클릭
3.  "사용자 지정 지침" 찾기
4.  이 가이드 상단에서 완전한 메모리 뱅크 지침을 복사하여 붙여넣기

#### .clinerules (프로젝트별)

1.  프로젝트 루트에 `.clinerules` 파일 생성
2.  이 가이드 상단에서 메모리 뱅크 지침을 복사하여 붙여넣기
3.  파일 저장
4.  Cline은 이 프로젝트에서 작업할 때 이러한 규칙을 자동으로 적용합니다.
