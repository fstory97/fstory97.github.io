1. 캐럿 소개

캐럿 프로젝트 소개

  • Cline의 모든 기능을 포함하며 100% 호환성을 유지하는 AI 코딩 파트너
  • '개인화된 AI 동반자'와 '강화된 시스템 프롬프트'를 통해 개발 경험을 향상

캐럿의 추가 기능

  • F07: 페르소나 시스템 - 사용자가 자신만의 AI 이름과 프로필 이미지를 등록하여 시각적으로 교감
  • F06: 강화된 프롬프트 시스템 - JSON 기반 동적 프롬프트와 Chatbot/Agent 모드 전환
  • F02: 4개 국어 지원 - UI, AI 채팅, 문서까지 완벽한 다국어(한/영/일/중) 환경
  • F05: 규칙 우선순위 시스템 - 다중 레이어 설정을 통한 커스터마이징
  • F11: 지식 패리티 시스템 - AI와 개발자가 동일한 지식 공유
  • F03: 브랜딩 및 UI 시스템 - B2B 솔루션을 위한 동적 브랜드 전환 (Caret/CodeCenter)
  • F10: 입력 이력 시스템 - 터미널과 같은 입력 이력 관리

2. 문제 정의

핵심 과제

강력한 오픈소스인 Cline 위에 구축된 프로젝트로서, 원본(Upstream)의 빠른 업데이트에 어떻게 지속적으로 대응하며 Caret만의 고유 가치를 유지하고 확장할 것인가?

참고 소스

  • Caret GitHub: github.com/aicoding-caret/caret
  • Cline GitHub: github.com/cline/cline

Caret 기능의 기술 영역

  • 페르소나 시스템 (F07)
  • 입력 이력 시스템 (F10)

3. 캐럿/클라인 구조 설명

핵심 아키텍처 사상: 최소 침습 (Minimal Invasion)

원본 Cline 코드를 최대한 보존하고, 외과수술처럼 정밀하게 필요한 부분만 연결하여 기능을 확장

성과

  • 수정 파일 85% 감소 (58개→9개)
  • 수정 라인 98% 감소 (수백→291)

디렉토리 기반 역할 분리

  • Backend: caret-src/
  • Frontend: webview-ui/src/caret/
  • Interface: proto/caret/

4.1. 1차 포크: 직접 수정의 시대와 실패한 백업 전략

접근법

  • Cline 초기 버전 위에서 필요한 기능을 원본 코드 58개 파일에 직접 추가
  • 3-Way 비교 방식 도입: cline-latest(원본), caret-main(과거), 작업 폴더(현재)를 비교하며 체계적으로 머징

실패한 아이디어

AI가 참고할 수 있도록 수정 전 파일을 .cline으로 백업

부작용

AI가 작업 중 기존 수정 파일을 다시 .cline으로 백업하면서,
Caret의 변경 소스가 백업 파일로 덮어씌워지는 등 혼란 가중

결국 이 방식은 폐기됨

4.2. 1차 머징: 고통 속에서 얻은 교훈과 새로운 전략

문제점 & 원인

  • '표준화 지옥': i18n 키 중복/순환 참조
  • AI의 지엽적 관점으로 전체 컨벤션 파악 어려움

터닝 포인트

  • 명확한 코딩 컨벤션 문서화의 중요성
  • F11: AI-개발자 지식 동기화 시스템 발상

실행 전략

  • Backend (Injection Model): Caret 서비스를 원본에 주입
  • Frontend (Base & Patch Model):
    1. Caret 코드를 기준(Base)으로 설정
    2. Cline의 기능 변경사항만 선별
    3. Caret 코드 위에 덮어쓰기(Patch)

4.3. 2차 머징: 아키텍처의 증명

도전 & 과정

Cline의 대규모 구조 변경으로 단순 머징이 불가능해지자, 새로운 Cline 버전 위에 '최소 침습' 아키텍처를 그대로 다시 이식

결과: 아키텍처의 유효성 증명

'최소 침습' 원칙 덕분에 Caret의 코드는 대부분 수정 없이 이식 가능했으며, 연결점(hook)만 다시 설정하여 성공적으로 업데이트를 완료했습니다.

5. 상세 해결 방법: 최소 침습 방법론의 구체적 실행

  • 5.1. 사상 및 설계 원칙
  • 5.2. 실행 방법

5.1. 사상 및 설계 원칙

사상

"원본은 그대로, 기능은 외부에서."
원본 코드를 '수정'의 대상이 아닌 '활용'의 대상으로 본다

원칙 1 (독립성)

Caret의 모든 기능은 caret-src, webview-ui/src/caret 등 약속된 독립 공간에서 구현

원칙 2 (최소 연결)

원본 코드 수정은 기능 활성화를 위한 연결점(hook) 역할의 1~3줄로 제한

원칙 3 (명시성)

모든 연결점에는 // CARET MODIFICATION 주석을 달아 수정 의도와 내용을 명확히 추적

5.2. 실행 방법

3-Way 비교 머징

cline-latest(원본), caret-main(과거 Caret), 작업 폴더(현재) 3개의 소스를 비교하여 통합

주석 전략

// CARET MODIFICATION: [수정 이유와 내용 요약] 형식으로 모든 수정 지점에 기록

백엔드 (Wrapper 패턴 예시)

/// In: caret-src/services/activation-wrapper.ts (Caret's new file)
export class ActivationWrapper {
  static activate(context) {
    console.log("✨ Caret features activated!");
    originalActivate(context);
  }
}

// In: src/extension.ts (Cline's original file)
export async function activate(context) {
  // CARET MODIFICATION: Use Caret's wrapper for activation
  ActivationWrapper.activate(context);
}

5.2. 실행 방법 (계속)

Protos (gRPC)

  • proto/cline 원본 파일을 절대 수정하지 않음, proto/caret에 Caret 전용 메시지와 서비스 정의
  • 충돌 회피 전략: Caret 고유의 필드 번호는 1000번 이상으로 지정 (Cline은 1~999)
  • enum 회피: enum 대신 string 또는 int32 사용

프론트엔드 (독립 컴포넌트 패턴)

// In: webview-ui/src/caret/components/PersonaSelector.tsx (Caret's new file)
export const PersonaSelector = () => {
  return <div className="persona-selector">...</div>;
};

export const Settings = () => (
  <div>
    <ClineSettings />
    <PersonaSelector />
  </div>
);

6.1. 전통적인 머징 방법론 대비

강점

  • Upstream 업데이트 시 머징 비용이 획기적으로 감소
  • 프로젝트의 유지보수성이 극대화
  • Caret만의 정체성과 로드맵을 명확하게 유지

약점

  • 초기 아키텍처 설계에 더 많은 고민 필요
  • Wrapper/DI 패턴으로 인해 코드 흐름 추적 시 한 단계를 더 거쳐야 하는 약간의 복잡성 존재
    (예: A → Wrapper → B 구조)

6.2. 오픈소스 생태계에 미치는 영향

새로운 협력 모델: Fork & Extend (포크 후 확장)

단순히 갈라지는 'Fork & Diverge'가 아닌, 원본과 함께 성장하는 모델

건강한 선순환 구조

  • 원본 발전 지속적 흡수
  • 독립적 가치 창출
  • 원본에 역기여(Contribute Back)

새로운 패러다임

  • AI 코드리뷰로 품질 통제
  • 빠른 fork/개선으로 다양한 시도
  • → SW 코드의 종 분화

6.3. AI 기반 SW 개발 방법론 측면

'표준화 지옥'의 교훈 & 해결책 (F11: 지식 동기화)

AI의 지엽적 관점 문제를 해결하기 위해, AI와 개발자가 동일한 코딩 컨벤션과 작업 절차(.caretrules)를 공유하는 '지식 동기화 시스템' 구축

결과: 개발 속도 18배 향상

AI에게 명확한 가이드라인을 제공하여 11일 만에 3~6개월 분량의 아키텍처 전환을 달성

AI 협업 모델 정립

전략/표준화(Human-Led) → 구현 가속(AI-Assisted) → 최종 검증(Human-Verified)

새로운 역량의 중요성

디버깅보다 설계, 모니터링, 검증이 중요 → SDLC 이해도 상승
