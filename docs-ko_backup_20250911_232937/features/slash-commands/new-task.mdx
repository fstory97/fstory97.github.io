title: "새 작업 명령"
sidebarTitle: "/newtask"

`/newtask`는 완벽한 개발자 핸드오프처럼 작동하는 슬래시 명령입니다. 전체 계획, 완료된 작업, 관련 파일 및 다음 단계와 같은 중요한 내용을 깔끔한 컨텍스트 창이 있는 새로운 작업으로 지능적으로 패키징합니다. 도구 호출, 문서 검색 및 구현 세부 정보와 같은 불필요한 정보는 모두 남겨둡니다.

이는 새로운 개발자를 프로젝트에 투입할 때 정확히 수행할 작업입니다. 이전의 모든 키 입력으로 압도하지 않고 작업을 계속하는 데 필요한 필수 컨텍스트를 제공합니다.

#### `/newtask` 슬래시 명령 사용

컨텍스트 창이 가득 차고 있지만 프로젝트가 완료되지 않은 경우:

-   채팅 입력 필드에 `/newtask`를 입력합니다.
-   Cline은 대화를 분석하고 전달할 컨텍스트의 정제된 버전을 제안합니다.
-   커밋하기 전에 대화를 통해 이 제안된 컨텍스트를 다듬을 수 있습니다.
-   만족하면 정제된 컨텍스트로 새 작업을 생성하는 버튼이 나타납니다.

#### 예시

저는 여러 단계가 있는 복잡한 구현을 작업할 때 `/newtask`를 정기적으로 사용합니다. 예를 들어, 10단계 프로세스 중 3단계를 완료했고 컨텍스트가 문서 스니펫, 파일 내용 및 상세한 논의로 이미 75% 가득 찬 경우.

그러한 통찰력을 잃거나 처음부터 시작하는 대신 `/newtask`를 사용하여 개별 도구 호출 및 연구 단계의 모든 불필요한 정보 없이 중요한 내용(주요 결정, 파일 변경 사항 및 현재까지의 진행 상황)을 추출하도록 Cline에 지시합니다.

저는 `/newtask`를 프로젝트에 합류하는 새로운 개발자라고 생각합니다. 그들에게 완료된 작업에 대한 완전한 이해, 관련 파일에 대한 인식, 도움이 될 수 있는 기타 컨텍스트 및 다음에 어디로 가야 할지를 제공해야 합니다.

#### 영감

다음은 `/newtask`를 사용하는 몇 가지 인기 있는 방법입니다.

-   Context7 MCP 서버를 사용하여 복잡한 API를 조사하여 컨텍스트를 문서로 채웁니다. 개념을 이해하면 `/newtask`를 사용하여 구현에 필요한 필수 지식만으로 새로 시작합니다.
-   여러 디버깅 시도 및 파일 탐색을 통해 어려운 버그의 근본 원인을 식별한 후 `/newtask`를 사용하여 솔루션을 포함하지만 실패한 모든 시도를 버리는 깨끗한 상태로 계속합니다.
-   클라이언트 논의가 여러 접근 방식을 탐색하고 마침내 한 방향으로 결정되면 `/newtask`를 사용하여 선택한 솔루션을 구현하는 데만 집중합니다.
-   여러 날에 걸쳐 진행되는 복잡한 프로젝트의 경우 논리적인 중단점에서 `/newtask`를 사용하여 진행 상황을 전달하면서 깔끔한 작업 공간을 유지합니다.
